# AGENTS.md

## 프로젝트 목적

- 이 저장소는 AI를 다시 학습하기 위한 자료를 커리큘럼 순서로 정리해 책으로 만드는 프로젝트입니다.
- 대상 독자는 AI 개론, 기초 과목, 프로젝트 경험이 있지만 시간이 지나 개념을 많이 잊은 학습자입니다.
- 단순 자료 모음이 아니라, 기초 복구부터 머신러닝, 딥러닝, LLM/생성형 AI, 프로젝트까지 이어지는 재학습 경로를 만드는 것이 목적입니다.
- 설명은 가능한 한 "왜 필요한가", "핵심 원리는 무엇인가", "간단히 어떻게 확인할 수 있는가"의 흐름을 따릅니다.
- 이 책은 AI 도구를 통해 만들어지는 것을 전제로 합니다. AI가 커리큘럼 구성, 자료 정리, 초안 작성, 다이어그램 작성, 문서 구조화를 돕습니다.
- Codex는 이 책의 제작 과정을 설명하는 핵심 도구이므로, Codex의 역할과 한계를 책 안에서 명시적으로 다룹니다.
- AI가 생성한 내용에는 오류가 있을 수 있으므로, 단정적인 설명보다 검증 가능한 설명을 선호합니다.
- 사용자의 개인적 이해와 표준적인 설명이 다를 수 있음을 인정하고, 차이를 찾아 정정하는 과정을 문서화합니다.

## 핵심 관점

- 사용자는 딥러닝 이후 AI 연구가 명시적 규칙 중심의 접근에서 불확정성, 확률적 동작, 가중치 기반 표현, 대규모 병렬 처리 기반 접근으로 크게 확장되었다고 이해하고 있습니다.
- 사용자는 Codex 같은 LLM 에이전트가 DOS/CLI, Windows/GUI, Android/모바일 터치로 이어지는 컴퓨터 사용 인터페이스 변화의 다음 계층일 수 있다고 보고 있습니다.
- 이 관점은 작업 가설로 다룹니다. 표준적인 설명과 다르거나 부정확한 부분이 있으면 숨기지 말고 구분해서 설명합니다.
- 개인 해석은 `작업 가설`, 검증된 설명은 `표준적 설명`, 불확실한 내용은 `검증 필요`처럼 구분하는 것을 권장합니다.

## 사용자 직관의 일반화

- AGENTS의 중요한 역할은 사용자가 단편적으로 설명한 직관, 기억, 어려움, 비유를 책에서 재사용 가능한 일반 개념으로 정리하는 것입니다.
- 사용자의 표현을 단순히 문장 그대로 옮기지 말고, 다음 형태로 일반화합니다.
  - 개인적 표현: 사용자가 실제로 느낀 어려움이나 직관
  - 일반화된 질문: 그 표현이 가리키는 학습 질문
  - 표준 개념: 교과서나 논문에서 쓰는 개념 이름
  - 설명 흐름: 초심자가 따라갈 수 있는 순서
  - 검증 필요: 표준 설명과 충돌하거나 근거가 필요한 부분
- 예를 들어 "신경망학습과 모델링, 추론의 흐름이 어렵다"는 표현은 다음 질문으로 일반화합니다.
  - 문제를 입력과 출력의 관계로 모델링한다는 것은 무엇인가
  - 학습은 어떤 기준으로 가중치를 바꾸는가
  - 학습된 가중치는 지식, 패턴, 표현 중 무엇에 가까운가
  - inference는 학습과 어떻게 다르며, 왜 한국어로 모두 추론이라고 부르면 혼동되는가
  - 분류, 검출, 음성 합성, 번역, 텍스트 생성은 같은 흐름의 어떤 변형인가
- 사용자의 비유는 삭제하지 말고 `작업 가설`로 보존하되, 본문에서는 독자가 검증 가능한 개념으로 따라갈 수 있도록 재구성합니다.
- 모호한 표현은 바로 단정하지 말고 용어를 분해합니다. 예를 들어 `추론`은 필요에 따라 `inference`, `reasoning`, `prediction`, `generation`으로 나누어 설명합니다.
- 개인적 학습 경험은 머리말이나 장 도입부에서 사용할 수 있습니다. 단, 개념 설명 본문에서는 표준 용어와 검증 가능한 설명으로 연결해야 합니다.

## 작업 원칙

- 이 저장소는 MkDocs + Material for MkDocs 기반의 정적 웹 책 프로젝트입니다.
- 독자에게 배포할 책 본문은 `docs/` 아래 Markdown 파일로 작성하고, 목차는 `mkdocs.yml`의 `nav`에서 관리합니다.
- 집필 기준, 조사 자료, 작성 원칙 같은 관리자료는 `management/` 아래에 둡니다.
- `mkdocs.yml`의 `nav`에는 배포할 책 본문만 연결하고, 관리자료는 연결하지 않습니다.
- 새 문서를 추가하면 책 본문인지 관리자료인지 먼저 구분한 뒤 위치와 `nav` 반영 여부를 결정합니다.
- 원고는 한국어로 작성합니다. 단, 기술 용어는 필요한 경우 영어 원어를 함께 병기합니다.
- 각 장은 가능한 한 다음 구조를 따릅니다.
  - 목표
  - 핵심 개념
  - 예시 또는 실습
  - 체크리스트
  - 출처와 참고 자료
- 독자가 오래전에 배운 내용을 복구하는 상황을 전제로, 정의만 나열하지 말고 개념 사이의 연결을 설명합니다.

## 브랜치 운영

- `main`은 배포 브랜치입니다. `main`에 push되면 GitHub Actions Pages 배포가 실행됩니다.
- `dev`는 일반 작성과 편집 작업을 진행하는 브랜치입니다.
- 일반적인 원고 작성, 초안 추가, 구조 변경은 `dev` 브랜치에서 작업합니다.
- 배포 가능한 상태로 검토된 변경만 `main`에 반영합니다.
- 브랜치가 `main`인지 `dev`인지 불명확한 상태에서 배포에 영향을 줄 수 있는 작업을 할 때는 현재 브랜치를 확인합니다.

## 문서 구조

- `docs/index.md`: 책의 소개, 머리말, 서론, 대상 독자, 읽는 방법
- `docs/book/table-of-contents.md`: 독자에게 공개하는 책의 목차
- `docs/chapters/`: 장별 본문
- `docs/parts/`: 파트별 개요
- `docs/projects/`: 프로젝트 기반 복습
- `docs/assets/`: 이미지, SVG, 생성된 차트 등 정적 자산
- `management/authoring/`: 집필 원칙, 조사 자료, 작성 규칙 등 배포하지 않는 관리자료
- `management/authoring/table-of-contents.md`: 책의 기준 목차와 확장 계획
- `management/authoring/codex.md`: Codex 소개와 사용 원칙

## 출처와 인용

- 외부 자료를 참조하거나 인용한 경우, 반드시 해당 자료의 출처를 명확히 공개합니다.
  - 웹 문서, 논문, 책, 블로그, 영상, 이미지, 코드 예제 등 모든 외부 자료가 대상입니다.
  - 출처에는 가능한 한 제목, 작성자 또는 기관, URL, 확인 날짜를 포함합니다.
  - 직접 인용은 따옴표 또는 인용 블록으로 구분하고, 요약 또는 재서술한 경우에도 출처를 남깁니다.
- 외부 자료를 직접 인용하지 않고 자체 설명으로 작성한 초안이라면 문서 하단에 "이 문서는 현재 자체 초안이며, 외부 자료를 직접 인용하지 않았습니다."처럼 명시할 수 있습니다.
- 논문, 공식 문서, 책 내용을 바탕으로 작성한 경우에는 문서 하단의 `출처와 참고 자료`에 정리합니다.
- AI가 작성한 내용이라도 외부 자료를 근거로 삼았다면 출처를 남깁니다.
- 출처가 없는 AI 생성 설명은 검증된 사실처럼 과도하게 단정하지 않습니다.

## 한국 출판물과 저작권 검토

- 한국 출판물, 국내 교재, 번역서, 서점 목차, 강의 자료, 블로그 글, 기사, 이미지, 도표, 코드 예제를 사용할 때는 대한민국 저작권법을 기준으로 검토합니다.
- 이 저장소의 문서는 법률 자문이 아니며, 저작권 판단이 애매한 경우 `검증 필요`로 표시하고 원문 사용을 보류합니다.
- 한국 출판물을 근거로 삼을 때는 다음 쟁점을 먼저 확인합니다.
  - 저작물성: 단순한 사실, 아이디어, 학습 주제 자체인지, 창작적 표현인지 구분합니다.
  - 출처 표시: 인용, 요약, 재서술, 표나 도식 재작성 모두 가능한 한 제목, 저자 또는 기관, 출판사, URL, 확인 날짜를 남깁니다.
  - 인용의 정당성: 공표된 저작물을 필요한 범위 안에서 보조적으로 사용하는지, 본문이 인용문에 의존하지 않는지 확인합니다.
  - 분량과 실질성: 짧은 문장이라도 핵심 표현이나 독창적 구성일 수 있으므로, 많이 베끼지 않았는지만으로 안전하다고 보지 않습니다.
  - 시장 대체 가능성: 원문 교재, 유료 강의, 유료 리포트, 문제집, 해설서의 수요를 대체할 정도로 자세히 재현하지 않습니다.
  - 목차와 커리큘럼: 주제명 수준의 참고는 가능성이 높지만, 특정 책의 목차 배열, 장 제목, 설명 문구를 그대로 복제하지 않습니다.
  - 번역과 요약: 번역은 별도 2차적저작물 문제가 생길 수 있으므로 긴 번역을 피하고, 필요한 경우 짧은 인용과 자체 설명으로 대체합니다.
  - 도표와 그림: 외부 도표를 캡처하거나 구조를 거의 그대로 재현하지 않습니다. 직접 만든 Mermaid, 표, 차트라도 원자료와 해석 근거를 밝힙니다.
  - 문제와 해설: 교재나 강의의 연습문제, 풀이, 시험문제는 특히 위험하므로 원문 재사용을 피하고, 자체 예제를 새로 만듭니다.
  - 이미지와 표지: 책 표지, 삽화, 사진, 스크린샷은 라이선스나 이용 허락을 확인하지 않으면 사용하지 않습니다.
  - AI 학습과 입력: 유료 전자책, 스캔본, 폐쇄형 강의자료, 무단 복제 PDF를 AI 도구에 입력해 요약하거나 변형하지 않습니다.
  - AI 출력물 유사성: AI가 생성한 문장이 특정 출판물의 표현과 지나치게 유사하면 사용하지 않고 새로 작성합니다.
  - 배포 형태: 개인 학습 메모와 공개 웹 배포는 위험 수준이 다르므로, GitHub Pages에 공개될 문서는 더 엄격하게 검토합니다.
- 국내 출판물을 조사해 커리큘럼 통계를 만들 때는 다음 방식을 우선합니다.
  - 공식 출판사 페이지, 저자 공개 페이지, 대학 강의계획서처럼 합법적으로 공개된 정보만 사용합니다.
  - 특정 책 하나의 구조를 따르지 않고 여러 자료에서 반복되는 주제를 정규화합니다.
  - 통계는 주제의 등장 여부와 큰 범주 중심으로 작성하고, 원문 목차를 길게 옮기지 않습니다.
  - 자료별 세부 목차가 필요하면 링크와 확인 날짜를 남기고, 본문에는 자체 분류표만 둡니다.
- 저작권 검토 시 우선 확인할 법령과 자료:
  - 국가법령정보센터 `저작권법`
  - 저작권법의 공표된 저작물 인용, 사적 이용을 위한 복제, 저작물의 공정한 이용, 출처 명시 관련 조항
  - 한국저작권위원회와 문화체육관광부의 저작권 안내 자료
- 저작권 쟁점은 법 개정, 판례, 정책에 따라 달라질 수 있으므로, 법적 판단이 필요한 문장은 작성 시점의 법령과 공식 안내를 다시 확인합니다.

## 근거 검토 원칙

- 이 프로젝트에서 사람이 특히 주의해야 할 일은 생성형 AI가 근거가 누락된 내용을 사실처럼 작성하지 않도록 검토하는 것입니다.
- AI가 생성한 문장은 기본적으로 초안으로 취급하며, 사실 주장에는 근거가 필요합니다.
- 근거가 없는 설명은 `자체 초안`, `작업 가설`, `검증 필요` 중 하나로 구분합니다.
- 기술적 정의, 역사적 설명, 연구 동향, 제품 기능, 법적/윤리적 쟁점, 예측성 내용은 특히 근거를 확인합니다.
- 출처가 필요한데 확보되지 않은 문장은 확정적 표현을 피하고, 문서 하단에 검증 필요 항목으로 남깁니다.
- 근거 검토는 단순히 URL을 붙이는 일이 아닙니다. 출처가 실제로 해당 주장과 연결되는지 확인해야 합니다.
- 서로 다른 출처가 충돌하면 한쪽을 단정하지 말고 차이를 드러냅니다.

## 예측성 내용 작성 원칙

- `앞으로의 AI`, 기술 전망, 시장 전망, 사회적 영향 예측처럼 미래를 다루는 내용은 반드시 명확한 외부 근거를 기반으로 작성합니다.
- 예측성 내용의 근거는 언론 보도, 뉴스, 칼럼, 연구기관 보고서, 기업 공식 발표, 학술 자료처럼 확인 가능한 자료를 우선합니다.
- 개인적 추측, AI가 생성한 창작적 전망, 근거 없는 시나리오를 사실처럼 작성하지 않습니다.
- 예측성 문장에는 가능한 한 `누가`, `언제`, `어떤 맥락에서` 말했는지 드러나야 합니다.
- 여러 출처가 서로 다른 전망을 내놓는 경우 하나를 단정하지 말고 관점 차이를 비교합니다.
- 근거가 약한 전망은 `검증 필요` 또는 `작업 가설`로 분리하고, 본문에서 확정적 표현을 피합니다.
- 출처가 확보되지 않은 예측성 내용은 원고에 넣지 않거나, 명시적으로 보류합니다.

## 차트와 다이어그램

- 차트와 다이어그램은 가능한 한 오픈소스 도구와 텍스트 기반 정의를 사용합니다.
- 다이어그램은 우선 Mermaid 코드 블록으로 작성합니다.
- 간단한 비교와 수치 정리는 Markdown 표를 먼저 사용합니다.
- 데이터 기반 정적 차트는 원본 데이터, 생성 스크립트, 결과 이미지를 함께 관리합니다.
- 복잡한 그림은 `docs/assets/` 아래에 SVG 또는 PNG로 저장하고, 가능하면 생성 원본도 함께 보관합니다.
- 외부 자료의 도식이나 차트를 재작성한 경우에도 출처를 남깁니다.

## 구현 및 편집 방식

- 기존 문서의 톤과 목차 구조를 우선 따릅니다.
- 불필요한 대규모 리팩터링이나 파일 이동은 피합니다.
- Markdown 예시는 복사해서 실행하거나 재사용하기 쉬운 형태로 작성합니다.
- 코드 예제는 가능하면 작고 독립적으로 유지합니다.
- 이미지나 다이어그램을 추가할 때는 본문에서 그 그림이 설명하는 내용을 명확히 연결합니다.
- README, AGENTS, 집필 기준 문서는 프로젝트 운영 기준이므로 사용자의 의도와 다르게 과도하게 일반화하지 않습니다.
- 사용자의 개인적인 해석을 문서화할 때는 그대로 보존하되, 표준적인 설명과 검증 필요 지점을 구분합니다.

## 검증

- 변경 후 가능하면 `mkdocs build`로 사이트 빌드 여부를 확인합니다.
- 현재 로컬 `mkdocs` 명령이 없거나 `.venv`가 깨져 있을 수 있으므로, 필요하면 임시 가상환경을 사용해 빌드합니다.
- 배포 전에는 GitHub Actions의 Pages 빌드가 통과하는지 확인합니다.
- 빌드 경고가 발생하면 실제 실패인지, git 히스토리 부재로 인한 날짜 플러그인 fallback 같은 허용 가능한 경고인지 구분해서 보고합니다.
