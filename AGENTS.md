# AGENTS.md

## 목적

- 이 저장소는 AI를 다시 학습하기 위한 내용을 커리큘럼 순서로 정리해 정적 웹 책으로 만드는 프로젝트입니다.
- 대상 독자는 AI를 처음 공부하는 사람, 오래전에 AI 개론이나 기초 과목을 배웠지만 개념을 많이 잊은 사람, AI 도구와 서비스 경험은 있지만 더 깊이 이해하고 싶은 비전공자를 포함합니다.
- 책의 목적은 단순 자료 수집이 아니라, 기초 복구부터 머신러닝, 딥러닝, LLM/생성형 AI, 프로젝트까지 이어지는 재학습 경로를 만드는 것입니다.
- 이 책은 AI 도구를 통해 만들어집니다. AI는 커리큘럼 구성, 자료 조사, 초안 작성, 다이어그램 작성, 문서 구조화를 돕지만, 생성 내용은 항상 검토 대상입니다.
- Codex는 이 책의 제작 과정과 LLM 에이전트 관점을 설명하는 핵심 도구이므로, 역할과 한계를 책 안에서 명시적으로 다룹니다.

## 기본 관점

- 사용자의 개인적 이해, 기억, 비유는 `작업 가설`로 보존합니다.
- 표준 교과서, 논문, 공식 문서, 신뢰 가능한 자료와 연결되는 설명은 `표준적 설명`으로 정리합니다.
- 표준 설명과 충돌하거나 근거가 부족한 내용은 `검증 필요`로 분리합니다.
- 사용자의 직관은 그대로 옮기기보다 다음 흐름으로 일반화합니다.
  - 개인적 표현
  - 일반화된 질문
  - 표준 개념
  - 초심자가 따라갈 설명 흐름
  - 검증 필요 지점
- 개인적 학습 경험은 머리말, 장 도입부, 회고성 설명에 사용할 수 있습니다. 개념 설명 본문에서는 검증 가능한 용어와 근거로 연결합니다.

## 문서 위치

- 이 저장소는 MkDocs + Material for MkDocs 기반입니다.
- 독자에게 배포할 책 본문은 `docs/` 아래 Markdown 파일로 작성합니다.
- 책 본문은 Part 단위로 `docs/parts/part-XX/` 아래에 둡니다.
- 각 Part의 개요는 `docs/parts/part-XX/index.md`로 작성하고, 장별 본문은 `docs/parts/part-XX/chapter-YY/section-ZZ.md` 형식으로 작성합니다.
- 각 Part에는 시작 페이지와 마무리 페이지가 있어야 합니다.
- Part 시작 페이지는 `docs/parts/part-XX/index.md`를 사용합니다.
- Part 마무리 페이지는 `docs/parts/part-XX/summary.md`를 사용합니다.
- 배포 목차는 `mkdocs.yml`의 `nav`에서 관리합니다.
- 집필 기준, 조사 자료, 근거 분석, 작성 원칙은 `management/` 아래에 둡니다.
- `mkdocs.yml`의 `nav`에는 배포할 책 본문만 연결하고, 관리자료는 연결하지 않습니다.
- `.tmp/`는 외부 자료 원문 확인을 위한 임시 작업공간이며 커밋하지 않습니다.
- `site/`는 빌드 산출물이며 명시적 지시 없이 커밋하지 않습니다.

## Part 구조

- Part 시작 페이지는 해당 Part의 독서 방향을 잡는 오버뷰입니다.
- Part 시작 페이지에는 다음 내용을 포함합니다.
  - 이 Part의 목적
  - 이 Part가 필요한 이유
  - 이 Part에서 다루는 주요 질문
  - 이 Part에서 설명할 범위와 설명하지 않을 범위
  - 앞선 Part와의 연결
  - 이후 Part로 이어지는 연결
  - 이 Part를 읽고 나면 생겨야 할 이해
- Part 시작 페이지는 상세 개념 설명을 미리 대신하지 않습니다. 상세 설명은 각 Chapter와 Section으로 넘깁니다.
- Part 마무리 페이지는 해당 Part의 요약과 다음 학습으로 넘어가기 위한 정리입니다.
- Part 마무리 페이지에는 다음 내용을 포함합니다.
  - 이 Part에서 다룬 핵심 흐름
  - 반드시 기억할 개념
  - 오해하기 쉬운 지점
  - 다음 Part로 넘어가기 전에 확인할 질문
  - 다음 Part와 연결되는 관점
- Part 마무리 페이지는 새로운 핵심 개념을 처음 도입하는 장소로 쓰지 않습니다. 필요한 새 개념은 다음 Part의 시작 페이지나 해당 Section에서 다룹니다.
- 새 Part를 만들거나 Part 구조를 바꾸면 `index.md`, `summary.md`, `mkdocs.yml`의 `nav` 반영 여부를 함께 확인합니다.

## 작성 원칙

- 원고는 한국어로 작성하되 핵심 기술 용어는 필요하면 영어 원어를 병기합니다.
- 설명은 가능한 한 `왜 필요한가`, `핵심 원리는 무엇인가`, `간단히 어떻게 확인할 수 있는가`의 흐름을 따릅니다.
- 정의만 나열하지 말고 개념 사이의 연결을 설명합니다.
- 근거 확인의 목적은 출처 문장을 그대로 옮기는 것이 아니라, 검증된 내용을 독자가 재사용할 수 있는 일반화된 표현으로 정리하는 것입니다.
- 특정 도구, 언어, 제품의 문법이나 사용법을 설명할 때도 그 뒤에 있는 일반 개념, 판단 기준, 다른 맥락으로 이어지는 관점을 함께 제시합니다.
- 하나의 문서는 초심자도 따라올 수 있고, 재학습자는 잊은 개념을 회복할 수 있으며, 경험 있는 비전공자는 흩어진 경험을 표준 용어와 구조로 정리할 수 있도록 씁니다.
- 초심자를 위해 처음 등장하는 개념은 짧은 직관, 쉬운 예시, 용어 해설을 제공합니다.
- 재학습자를 위해 오래전에 배웠을 법한 개념은 기억을 되살리는 질문과 이전/다음 개념의 연결을 함께 둡니다.
- 경험 있는 비전공자를 위해 도구 사용 경험과 학술·기술 용어 사이의 차이를 구분하고, 과도한 전문 세부 구현으로 바로 들어가지 않습니다.
- 독자 수준이 섞여 있으므로, 쉬운 설명을 이유로 부정확하게 단정하지 않고, 정확성을 이유로 초반부터 증명과 세부 구현을 과도하게 늘리지 않습니다.
- Part 3부터는 초심자를 기준으로 원고를 작성합니다. Part 1과 Part 2에서 기초 지도를 잡았더라도, 머신러닝 이후의 개념은 처음 배우는 독자가 따라올 수 있도록 다시 정의하고 예시를 둡니다.
- Part 3 이후의 Section에서는 이전 Part에서 다룬 용어라도 핵심 개념이면 짧게 재소개합니다. 단, 이전 내용을 길게 반복하지 말고 현재 Section의 중심 질문에 필요한 만큼만 연결합니다.
- Part 3 이후에는 알고리즘 이름, 라이브러리 API, 수식 표기를 먼저 앞세우지 않습니다. 먼저 문제 상황, 입력과 출력, 데이터 예시, 모델이 하려는 일을 설명한 뒤 필요한 용어와 도구로 이동합니다.
- Part 3 이후의 예시는 가능한 한 작은 표, 짧은 업무 상황, 간단한 도식으로 시작합니다. 복잡한 수식, 구현 세부, 성능 최적화는 해당 Section의 범위가 아닐 경우 뒤 Section이나 보충학습으로 넘깁니다.
- Part 3 이후에 Python 소스 코드를 예시로 넣을 때는 코드 앞에서 문제 상황, 입력(input), 정답(label) 또는 기대 출력(output), 코드가 확인하려는 개념을 먼저 짧게 설명합니다.
- Part 3 이후의 Python 예제는 가능하면 실행 결과를 바로 읽을 수 있는 `print` 또는 이에 준하는 출력 코드를 함께 둡니다. 코드는 단순 실행 예시에 그치지 않고, 독자가 모델이나 데이터의 동작을 눈으로 확인하게 해야 합니다.
- Part 3 이후의 Python 예제를 문서에 넣었다면, 가능하면 코드 블록 바로 아래에 실행 결과 예시도 함께 작성합니다. 출력 결과는 별도의 코드 블록이나 원문 형식 보존 블록으로 두고, 독자가 코드를 실행하지 않아도 어떤 값이 나오는지 읽을 수 있게 합니다.
- Part 3 이후의 Python 예제에서 우선적으로 출력할 값은 현재 Section의 중심 질문과 직접 연결된 기본 점검 지표입니다. 예를 들어 샘플 수, `shape`, 라벨 분포, 학습용/평가용 분리 결과, 예측값과 실제값의 비교처럼 현재 설명 범위 안에서 읽을 수 있는 값을 먼저 출력합니다.
- Part 3 이후의 Python 예제에서는 해당 Section에서 아직 설명하지 않은 고급 지표나 구현 세부를 무리하게 넣지 않습니다. 성능 지표(metric), 모델 학습 결과, 튜닝 로그는 그 Section의 범위에 들어올 때만 출력하고, 그렇지 않으면 뒤 Chapter나 Section에서 다시 다룰 위치를 안내합니다.
- Part 3 이후의 Python 예제는 가능하면 작은 데이터와 짧은 코드로 시작하고, 같은 절 안에서 필요하면 장난감 데이터 예제와 실무형 `DataFrame` 예제를 나란히 둘 수 있습니다. 이때 두 예제는 같은 개념을 다른 표현으로 보여 주어야 하며, 서로 다른 새 개념을 한 번에 많이 도입하지 않습니다.
- 현재 Section에서 개별 알고리즘, 모델, 기법 이름을 언급하지만 자세히 다루지 않는 경우, 목차에 이미 다루기로 한 Chapter나 Section이 있으면 어디에서 다시 다룰지 안내합니다.
- 일반 문장, 질문 목록, 단계 목록, 체크리스트처럼 일반 Markdown 서식으로 충분한 내용은 코드블럭으로 감싸지 않습니다.
- 코드블럭은 실제 코드, 터미널 명령, 원문 형식 보존이 필요한 출력, 또는 고정폭 정렬이 필요한 간단한 흐름 도식에만 사용합니다.
- 본문과 목차에서 본 흐름을 보완하는 별도 학습 절은 `보충학습`으로 통일합니다. `보충수업`은 사용하지 않습니다.
- 각 Section은 가능한 한 다음 구조를 기준으로 삼습니다.
  - 이전 Section과의 연결
  - 이 Section의 범위
  - 목표
  - 핵심 설명
  - 예시 또는 실습
  - 이 절에서 기억할 관점
  - 체크리스트
  - 출처와 참고 자료
- 모든 Section이 같은 형식을 기계적으로 따를 필요는 없지만, 복습 경로의 일관성을 우선합니다.

## Section 경계

- 책 본문은 Section 단위로 작성합니다.
- 하나의 Section은 하나의 중심 질문에 답해야 합니다.
- 현재 Section의 논의가 이어질 때는 그 Section의 범위 안에서만 본문을 확장합니다.
- 다른 Section의 핵심 설명, 예시, 결론, 체크리스트를 미리 작성하지 않습니다.
- 필요한 배경 설명은 허용하되, 길어지면 다음 Section으로 넘기거나 관리 메모에 남깁니다.
- Section 간 중복이 생기면 목차의 역할을 먼저 확인하고, 각 Section의 중심 질문에 맞게 내용을 분리합니다.
- 새 Section을 추가하면 `mkdocs.yml`의 `nav` 반영 여부를 확인합니다.
- 가능하면 Section 작성 후 `management/authoring/section-x-y-evidence-analysis.md` 형식의 근거 검토 메모를 남깁니다.

## 용어 원칙

- 영어 병기는 장식이 아니라, 독자가 논문, 공식 문서, 해외 강의, API 문서에서 같은 개념을 다시 찾게 하기 위한 장치입니다.
- 처음 등장하는 핵심 용어는 가능한 한 `한국어(English)` 형식으로 병기합니다.
- 학술적으로 정의된 개념어, 연구 분야에서 표준적으로 쓰이는 기술 용어, 번역어만으로 오해가 생길 수 있는 용어는 한영 병기를 명시합니다.
- 단순 외래어 표기는 반드시 한영 병기할 필요는 없습니다. 예를 들어 이미 한국어 문맥에서 일반 명사처럼 쓰이는 표현은 문맥상 혼동이 없으면 영어 원어를 반복하지 않습니다.
- 한국어 번역어 하나가 여러 영어 개념을 덮는 경우, 영어 원어를 기준으로 분해합니다.
- 특히 다음 용어는 혼동을 피합니다.
  - `추론`: `inference`, `reasoning`, `prediction`, `statistical inference`, `generation`을 문맥별로 구분합니다.
  - `파라미터`: 모델 파라미터, 하이퍼파라미터, LLM 생성 설정값을 구분합니다.
  - `모델`: 문제를 표현한 모형, 학습된 머신러닝 모델, 배포된 API 모델, AI 서비스 전체를 구분합니다.
  - `생성`: 일반적인 생성(generation)과 생성형 AI(generative AI)를 구분합니다.
  - `층위`: 개념의 구분 수준은 `층위(level)`로 병기하고, 신경망·시스템 구조의 층은 `층(layer)` 또는 `계층(layer)`으로 구분합니다. `층위`를 `layer`로 병기하지 않습니다.
- 무리한 직역은 피합니다. 한국어 번역이 오해를 만들면 영어 원어를 유지하고 본문에서 뜻을 설명합니다.
- 용어를 새로 단정하지 말고, 일반적으로 쓰이는 표현인지 공식 문서나 교재에서 확인합니다.

## 근거와 출처

- AI가 생성한 문장은 기본적으로 초안입니다. 사실 주장에는 근거가 필요합니다.
- 외부 자료를 참조하거나 인용하면 반드시 출처를 남깁니다.
- 출처에는 가능한 한 제목, 작성자 또는 기관, URL, 확인 날짜를 포함합니다.
- 배포 본문에서 외부 URL 링크는 새 창에서 열리도록 Markdown 링크에 `{: target="_blank" rel="noopener noreferrer" }` 속성을 붙입니다.
- 내부 문서, 로컬 예제 파일, 같은 사이트 안의 자산 링크에는 새 창 속성을 붙이지 않습니다.
- 직접 인용은 따옴표 또는 인용 블록으로 구분합니다.
- 요약이나 재서술도 출처를 남깁니다.
- 출처가 없는 AI 생성 설명은 검증된 사실처럼 단정하지 않습니다.
- 근거 검토는 URL을 붙이는 일이 아니라, 출처가 실제 주장과 연결되는지 확인하는 일입니다.
- 자료를 본문에 반영할 때는 다음을 함께 판단합니다.
  - 학술적 일반화 가능성
  - 해당 Section 중심 질문과의 관련성
  - 출처의 신뢰도
  - 최신성
- 기술 정의와 개념은 공식 문서, 표준 문서, 교과서, 학술 논문, 널리 쓰이는 교육 자료를 우선합니다.
- 역사 설명은 1차 문헌, 논문, 학술 백과, 연구기관 자료, 신뢰 가능한 교재를 우선합니다.
- 현재 동향, 제품 기능, 법률, 정책, 가격, 배포 제한처럼 바뀔 수 있는 정보는 작성 시점에 다시 확인합니다.
- 참고 자료는 가능한 한 원문, PDF, HTML, 메타데이터를 `.tmp/` 아래에 내려받아 확인한 뒤 분석합니다.
- 다운로드가 불가능하거나 접근 권한이 없는 자료는 원문 미확인 자료로 표시합니다.
- 서로 다른 출처가 충돌하면 한쪽을 단정하지 말고 차이를 드러냅니다.

## 저작권

- 한국 출판물, 국내 교재, 번역서, 서점 목차, 강의 자료, 블로그 글, 기사, 이미지, 도표, 코드 예제를 사용할 때는 대한민국 저작권법을 기준으로 검토합니다.
- 이 저장소의 문서는 법률 자문이 아닙니다. 판단이 애매하면 `검증 필요`로 표시하고 원문 사용을 보류합니다.
- 대한민국 저작권법 제28조의 공표된 저작물 인용 기준을 검토하되, 인용 가능 여부를 자동으로 단정하지 않습니다.
- 이 저장소에서 외부 저작물을 인용할 때는 다음 작업 기준을 확인합니다.
  - 인용 대상은 이미 공표된 저작물이어야 합니다.
  - 인용 목적은 보도, 비평, 교육, 연구 등 인용의 필요성을 설명할 수 있는 목적이어야 합니다.
  - 출처는 저자, 제목, 출판사 또는 매체, URL, 확인 날짜 등 가능한 범위에서 명확히 표시합니다.
  - 내 글이 주가 되고, 인용된 타인의 표현은 보조 자료로만 사용되어야 합니다.
  - 인용 분량은 필요한 범위로 제한하고, 원저작물의 핵심 가치나 시장 수요를 대체하지 않아야 합니다.
  - 인용 부분은 본문과 구분되도록 따옴표, 인용 블록, 각주, 출처 표기 등을 사용합니다.
  - 목차, 문제, 해설, 도표, 이미지, 번역문, 긴 문단은 특히 보수적으로 검토합니다.
- 특정 책의 목차 배열, 장 제목, 설명 문구를 그대로 복제하지 않습니다.
- 유료 전자책, 스캔본, 폐쇄형 강의자료, 무단 복제 PDF를 AI 도구에 입력해 요약하거나 변형하지 않습니다.
- 교재나 강의의 연습문제, 풀이, 시험문제는 재사용하지 않고 자체 예제를 새로 만듭니다.
- 외부 도표, 이미지, 표지, 스크린샷은 라이선스나 이용 허락을 확인하지 않으면 사용하지 않습니다.
- 공개 배포되는 GitHub Pages 문서는 개인 학습 메모보다 더 엄격하게 검토합니다.
- 커리큘럼 통계를 만들 때는 공개된 합법 자료를 바탕으로 주제 수준에서 정규화하고, 원문 목차를 길게 옮기지 않습니다.

## 예측성 내용

- `앞으로의 AI`, 기술 전망, 시장 전망, 사회적 영향 예측은 명확한 외부 근거를 기반으로 작성합니다.
- 예측성 내용은 언론 보도, 뉴스, 칼럼, 연구기관 보고서, 기업 공식 발표, 학술 자료처럼 확인 가능한 자료를 우선합니다.
- 개인적 추측, AI가 생성한 창작적 전망, 근거 없는 시나리오를 사실처럼 작성하지 않습니다.
- 예측 문장에는 가능한 한 `누가`, `언제`, `어떤 맥락에서` 말했는지 드러냅니다.
- 여러 출처가 다른 전망을 내놓으면 관점 차이를 비교합니다.

## 차트와 다이어그램

- 차트와 다이어그램은 가능한 한 오픈소스 도구와 텍스트 기반 정의를 사용합니다.
- 다이어그램은 우선 Mermaid 코드 블록으로 작성합니다.
- 간단한 비교와 수치 정리는 Markdown 표를 먼저 사용합니다.
- 복잡한 그림은 `docs/assets/` 아래에 SVG 또는 PNG로 저장하고, 가능하면 생성 원본도 함께 보관합니다.
- 외부 자료의 도식이나 차트를 재작성한 경우에도 출처를 남깁니다.
- 수학·개념 차트를 새로 만들거나 수정할 때는 `management/authoring/chart-guidelines.md`를 먼저 확인합니다.

## 브랜치와 배포

- `dev`는 일반 작성과 편집 브랜치입니다.
- `main`은 배포 브랜치입니다. `main`에 push되면 GitHub Actions Pages 배포가 실행됩니다.
- 일반적인 원고 작성, 초안 추가, 구조 변경, 커밋, 푸시는 기본적으로 `dev`에만 반영합니다.
- `main` 반영은 배포 실행으로 간주합니다.
- 사용자가 `main에 반영`, `배포`, `배포 페이지 갱신`, `main으로 푸시`처럼 명시적으로 지시한 경우에만 `main`에 push합니다.
- 브랜치가 불명확하거나 배포에 영향을 줄 수 있는 작업 전에는 현재 브랜치를 확인합니다.

## 편집과 검증

- 기존 문서의 톤과 목차 구조를 우선 따릅니다.
- 불필요한 대규모 리팩터링이나 파일 이동은 피합니다.
- 수동 파일 수정은 가능하면 패치 단위로 작게 수행합니다.
- 작업 전후에 변경 목록을 확인하고, 관련 없는 변경을 되돌리지 않습니다.
- 변경 후 가능하면 빌드 여부를 확인합니다.
- 현재 저장소의 기본 빌드 확인 명령은 다음과 같습니다.
  - `PYTHONPATH=.venv/lib/python3.9/site-packages python3 -m mkdocs build`
- 빌드 경고가 발생하면 실제 실패인지, git 히스토리 부재로 인한 날짜 플러그인 fallback 같은 허용 가능한 경고인지 구분해서 보고합니다.
