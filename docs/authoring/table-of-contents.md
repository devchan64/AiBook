# 기준 목차

이 문서는 책의 기준 목차입니다. 실제 장 제목과 순서는 학습을 진행하면서 바뀔 수 있지만, 큰 흐름은 유지합니다.

현재 기준 흐름은 다음과 같습니다.

```mermaid
flowchart LR
  A[AI 개론과 지형도]
  B[기초 복구]
  C[머신러닝]
  D[딥러닝]
  E[LLM과 생성형 AI]
  F[프로젝트]

  A --> B --> C --> D --> E --> F
```

## 문서 상태

- 상태: 초안
- 목적: 책의 큰 구조와 확장 기준 정의
- 적용 범위: `docs/` 원고와 `mkdocs.yml`의 `nav`
- 변경 원칙: 참고 목차를 그대로 복사하지 않고 책의 목적에 맞게 재배치

## 상태 표기

각 장은 다음 상태 중 하나로 관리합니다.

| 상태 | 의미 |
| --- | --- |
| 후보 | 목차에는 있지만 아직 문서화하지 않은 주제 |
| 초안 | AI 도구로 기본 설명을 작성한 상태 |
| 검토 필요 | 외부 자료 확인, 수식 검증, 코드 실행, 사실 확인이 필요한 상태 |
| 보강 중 | 사람이 방향을 정했고 AI가 자료와 예시를 추가하는 상태 |
| 안정 | 현재 수준에서 읽을 수 있고 출처와 검증 기준이 정리된 상태 |

이 책은 AI 도구로 만들어지므로 `초안`과 `안정`을 엄격히 구분합니다. 그럴듯한 설명이 있어도 출처, 예제, 검증 질문이 없으면 안정 상태로 보지 않습니다.

## 근거 검토 기준

생성형 AI는 근거가 누락된 내용을 자연스럽게 사실처럼 작성할 수 있습니다. 이 책에서 사람의 핵심 역할은 그 문장이 실제 근거를 갖는지 확인하는 것입니다.

| 문장 유형 | 필요한 처리 |
| --- | --- |
| 개념 정의 | 교과서, 공식 문서, 신뢰할 수 있는 강의나 문서로 확인 |
| 역사적 설명 | 연도, 인물, 사건을 확인 가능한 출처로 검증 |
| 제품 기능 | 공식 문서나 릴리스 노트로 확인 |
| 연구 동향 | 논문, 연구기관 보고서, 학술 자료로 확인 |
| 예측과 전망 | 언론, 뉴스, 칼럼, 공식 발표, 보고서 기반으로만 작성 |
| 개인 해석 | `작업 가설`로 표시하고 표준 설명과 구분 |

근거가 없거나 약한 문장은 `검토 필요` 상태로 둡니다. 출처 링크가 있더라도 해당 출처가 실제로 본문 주장을 뒷받침하는지 확인해야 합니다.

## 설계 원칙

- 먼저 커리큘럼의 기준을 잡고, 그 기준에 맞춰 자료를 수집합니다.
- AI는 자료 수집, 정리, 초안 작성, 비교, 다이어그램 작성을 돕습니다.
- 사람은 방향 설정, 오류 검토, 해석 보정, 최종 판단을 담당합니다.
- 개인적인 작업 가설과 표준적인 설명을 구분합니다.
- 틀린 내용이 발견되면 감추지 않고 정정합니다.
- 개론에서 모든 것을 깊게 설명하지 않습니다. 개론은 뒤에서 다시 배울 내용을 연결하는 지도 역할을 합니다.
- 각 장은 가능하면 `목표`, `핵심 개념`, `예시 또는 실습`, `체크리스트`, `출처와 참고 자료`를 포함합니다.
- 한 장은 하나의 중심 질문에 답하도록 작성합니다.
- 최신 기술이나 논쟁적인 해석은 `표준적 설명`, `작업 가설`, `검증 필요`를 구분합니다.

처음부터 모든 장을 채우지 않습니다. 재학습 경로를 빠르게 만들기 위해 다음 순서로 작성합니다.

| 우선순위 | 작업 | 이유 |
| --- | --- | --- |
| 1 | AI 개론과 지형도 | 이후 모든 장의 위치를 잡고, 사용자의 직관을 일반 개념으로 바꾸는 지도 역할 |
| 2 | 기초 복구 | 수학, Python, 데이터 처리의 빈틈을 먼저 줄임 |
| 3 | 머신러닝 핵심 | 학습, 평가, 일반화 개념을 회복 |
| 4 | 딥러닝 핵심 | 가중치, 역전파, GPU, 표현 학습을 정리 |
| 5 | LLM과 생성형 AI | 현재 관심사와 실무 적용으로 연결 |
| 6 | 프로젝트 | 이해한 내용을 산출물로 검증 |

각 우선순위 안에서는 개념 설명을 먼저 쓰고, 실습과 사례는 뒤따라 붙입니다.

## Part 1. AI 개론과 지형도

역할: 세부 기술을 깊게 들어가기 전에 전체 지형을 다시 잡습니다. "AI란 무엇인가"에서 시작하되, 단순 입문서식 나열이 아니라 이후 파트로 이어지는 연결 지도를 만듭니다.

Part 1은 공개 교재와 대학 강의에서 반복되는 `AI 개론`, `탐색`, `문제 해결`, `불확실성`, `학습` 축을 먼저 반영합니다. 그 위에서 사용자의 직관을 일반화합니다. 특히 "신경망학습, 모델링, 추론의 흐름이 어렵다"는 경험을 다음 질문으로 바꿉니다.

- 사람이 규칙을 쓰는 방식과 데이터에서 규칙을 배우는 방식은 무엇이 다른가
- 문제를 입력과 출력의 관계로 모델링한다는 것은 무엇인가
- 학습은 무엇을 바꾸고, inference는 무엇을 계산하는가
- 가중치는 지식, 패턴, 표현 중 무엇에 가까운가
- 이미지 인식, 객체 검출, 음성 합성, 번역, 텍스트 생성은 같은 딥러닝 흐름의 어떤 변형인가
- LLM은 이 흐름 위에서 무엇을 이어받고, 무엇을 새로 확장했는가

## 확장 규칙

새 장을 추가할 때는 다음을 확인합니다.

- 어느 Part에 속하는가
- 앞 장의 어떤 개념을 이어받는가
- 뒤 장의 어떤 개념을 준비하는가
- 자체 초안인지, 외부 자료 기반인지
- `mkdocs.yml`의 `nav`에 반영해야 하는가
- 차트나 다이어그램이 필요한가
- 검증 필요 목록에 추가할 질문이 있는가
- 예측성 내용이라면 명확한 외부 근거가 확보되었는가
- 사실 주장마다 근거가 있거나, 근거가 없는 경우 상태가 명확히 표시되었는가

## 파일 배치 규칙

- 파트 개요는 `docs/parts/` 아래에 둡니다.
- 장별 본문은 `docs/chapters/` 아래에 둡니다.
- 프로젝트 문서는 `docs/projects/` 아래에 둡니다.
- 집필 기준, 운영 원칙, 도구 설명은 `docs/authoring/` 아래에 둡니다.
- 이미지, SVG, 생성된 차트는 `docs/assets/` 아래에 둡니다.
- 새 파일을 만들면 `mkdocs.yml`의 `nav` 반영 여부를 반드시 확인합니다.
