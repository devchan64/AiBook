## 작성 판단

- 기존 P2-8.2가 리스트, 딕셔너리, 반복을 모두 포함해 길어졌기 때문에 딕셔너리 내용을 P2-8.3으로 분리했다.
- 사용자는 딕셔너리에서 범용적인 맵 성능 경험을 갖고 있다고 했다. 본문에서는 성능을 절대화하지 않고, “키로 값을 찾는 범용적인 mapping 구조”라는 기준 중심 표현으로 보강했다.
- 설정값, 라벨 맵, 샘플 ID, 컬럼 설명 예시를 사용해 키로 값을 찾는 패턴을 드러냈다.
- 단어 카운트처럼 반복하면서 딕셔너리에 값을 누적하는 예시는 P2-8.4로 재배치했다.
- Python 딕셔너리를 근거로 삼되, 본문에서는 key-value mapping이라는 일반 구조로 정리했다.
- 사용자는 딕셔너리와 object의 차이를 의식하고 있다. 본문에서는 JSON 객체나 일부 언어의 object 표현과 모양이 비슷할 수 있음을 인정하되, Python 딕셔너리는 key-value mapping으로 우선 설명했다.
- 일반적인 mapping 설명과 Python 딕셔너리 정의를 구분해 표기했다.
- 사용자는 딕셔너리를 보면 해시맵인지, 맵트리인지, 멀티인덱스인지 궁금해한다고 했다. Python 공식 문서로 직접 확인 가능한 표현은 mapping type, hashable key, insertion order, key-value pair, unique key이다. 본문에서는 “사용 관점에서는 매핑, 구현 관점에서는 해시 기반 구조”로 일반화하되, C++ STL의 `std::map`, `std::unordered_map`, multi-index와의 직접 등가 표현은 피했다.
- 사용자는 C/C++ STL 자료구조보다 대용량 데이터 접근이 쉽게 느껴진다고 했다. 본문에는 사용 편의성 체감은 인정하되, 대용량 데이터의 만능 구조라고 단정하지 않고 데이터베이스, pandas, 별도 인덱스 구조 검토가 필요할 수 있다고 제한했다.

## 본문에 반영한 안전한 설명

- 딕셔너리는 키와 값의 묶음이며, JSON 객체, 설정 파일, API 응답, 메타데이터를 읽는 관점으로 확장된다.
- 딕셔너리는 객체(object)와 모양이 비슷할 수 있지만, Python에서 object는 더 넓은 개념이고 딕셔너리는 mapping 구조다.
- 리스트는 위치로, 딕셔너리는 키로 값을 찾는다는 차이를 설명할 수 있다.
- 딕셔너리는 hashable key를 값에 연결하는 mapping 구조다.
- 딕셔너리는 해시맵(hash map)과 연결해 이해할 수 있지만, 초반에는 키로 값을 찾는 매핑(mapping) 구조로 먼저 설명한다.
- Python 딕셔너리의 키는 hashable해야 하므로, 키는 값을 찾는 안정적인 기준이어야 한다.
- Python 딕셔너리는 삽입 순서를 보존하지만, 키 기준 정렬 트리로 설명하지 않는다.
- 하나의 딕셔너리는 하나의 키로 값을 찾는 매핑이며, multi-index 구조로 단정하지 않는다.
- 같은 키에 값을 다시 저장하면 기존 값은 대체된다. 한 키에 여러 값을 모으려면 값 쪽에 리스트 같은 구조를 넣는다.
- 딕셔너리는 라벨 맵, 설정값, 메타데이터처럼 이름으로 값을 찾는 데이터에 적합하다.
- 키별 카운트는 딕셔너리와 반복이 결합된 패턴으로 P2-8.4에서 다룬다.
- 키가 없을 수 있는 상황에서는 `get()`을 사용할 수 있다.
