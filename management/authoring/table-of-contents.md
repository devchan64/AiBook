# 기준 목차

이 문서는 책의 기준 목차입니다. 실제 장 제목과 순서는 학습을 진행하면서 바뀔 수 있지만, 큰 흐름은 유지합니다.

현재 기준 흐름은 다음과 같습니다.

```mermaid
flowchart LR
  A[AI 개론과 지형도]
  B[기초 복구]
  C[머신러닝]
  D[딥러닝]
  E[LLM과 생성형 AI]
  F[프로젝트]

  A --> B --> C --> D --> E --> F
```

## 문서 상태

- 상태: 초안
- 목적: 책의 큰 구조와 확장 기준 정의
- 적용 범위: `docs/` 원고와 `mkdocs.yml`의 `nav`
- 변경 원칙: 참고 목차를 그대로 복사하지 않고 책의 목적에 맞게 재배치

## 상태 표기

각 장은 다음 상태 중 하나로 관리합니다.

| 상태 | 의미 |
| --- | --- |
| 후보 | 새로 제안되었지만 아직 기준 목차로 채택하지 않은 주제 |
| 확정 | 현재 기준 목차로 채택했지만 아직 본문이 작성되지 않은 주제 |
| 초안 | AI 도구로 기본 설명을 작성한 상태 |
| 검토 필요 | 외부 자료 확인, 수식 검증, 코드 실행, 사실 확인이 필요한 상태 |
| 보강 중 | 사람이 방향을 정했고 AI가 자료와 예시를 추가하는 상태 |
| 안정 | 현재 수준에서 읽을 수 있고 출처와 검증 기준이 정리된 상태 |

`확정`은 목차 채택 상태일 뿐, 내용이 검증되었다는 뜻이 아닙니다. 이 책은 AI 도구로 만들어지므로 `초안`과 `안정`을 엄격히 구분합니다. 그럴듯한 설명이 있어도 출처, 예제, 검증 질문이 없으면 안정 상태로 보지 않습니다.

## 근거 검토 기준

생성형 AI는 근거가 누락된 내용을 자연스럽게 사실처럼 작성할 수 있습니다. 이 책에서 사람의 핵심 역할은 그 문장이 실제 근거를 갖는지 확인하는 것입니다.

| 문장 유형 | 필요한 처리 |
| --- | --- |
| 개념 정의 | 교과서, 공식 문서, 신뢰할 수 있는 강의나 문서로 확인 |
| 역사적 설명 | 연도, 인물, 사건을 확인 가능한 출처로 검증 |
| 제품 기능 | 공식 문서나 릴리스 노트로 확인 |
| 연구 동향 | 논문, 연구기관 보고서, 학술 자료로 확인 |
| 예측과 전망 | 언론, 뉴스, 칼럼, 공식 발표, 보고서 기반으로만 작성 |
| 개인 해석 | `작업 가설`로 표시하고 표준 설명과 구분 |

근거가 없거나 약한 문장은 `검토 필요` 상태로 둡니다. 출처 링크가 있더라도 해당 출처가 실제로 본문 주장을 뒷받침하는지 확인해야 합니다.

## 설계 원칙

- 먼저 커리큘럼의 기준을 잡고, 그 기준에 맞춰 자료를 수집합니다.
- AI는 자료 수집, 정리, 초안 작성, 비교, 다이어그램 작성을 돕습니다.
- 사람은 방향 설정, 오류 검토, 해석 보정, 최종 판단을 담당합니다.
- 개인적인 작업 가설과 표준적인 설명을 구분합니다.
- 틀린 내용이 발견되면 감추지 않고 정정합니다.
- 개론에서 모든 것을 깊게 설명하지 않습니다. 개론은 뒤에서 다시 배울 내용을 연결하는 지도 역할을 합니다.
- 각 장은 가능하면 `목표`, `핵심 개념`, `예시 또는 실습`, `체크리스트`, `출처와 참고 자료`를 포함합니다.
- 한 장은 하나의 중심 질문에 답하도록 작성합니다.
- 한 Section은 하나의 중심 질문에 답하도록 작성하며, 논의가 이어져도 다른 Section의 핵심 내용을 침범하지 않습니다.
- 다른 Section에서 다룰 내용이 필요하면 현재 Section에는 짧은 연결 문장만 두고, 본문 확장은 해당 Section에서 진행합니다.
- 최신 기술이나 논쟁적인 해석은 `표준적 설명`, `작업 가설`, `검증 필요`를 구분합니다.
- 핵심 기술 용어는 영어 병기를 검토합니다. 병기는 장식이 아니라 논문, 공식 문서, 해외 강의, API 문서에서 같은 개념을 다시 찾기 위한 학습 연결 장치입니다.
- 한국어 번역어가 여러 영어 개념을 덮는 경우 원어를 구분합니다. 예를 들어 `추론`은 문맥에 따라 `inference`, `reasoning`, `prediction`을 나누어 씁니다.

## 목차 카테고리 체계

이 책의 기본 구조는 `Part > Module > Chapter > Section`으로 관리합니다.

| 수준 | 의미 | 사용 기준 |
| --- | --- | --- |
| Part | 책의 가장 큰 학습 영역 | AI 개론, 기초 복구, 머신러닝처럼 큰 흐름을 나눔 |
| Module | Part 안의 주제 묶음 | 여러 Chapter를 하나의 학습 흐름으로 묶음 |
| Chapter | 본문을 묶는 주제 단위 | 한 장이 하나의 중심 질문에 답함 |
| Section | 실제 원고 파일 단위 | 장 안에서 개념, 예시, 실습, 검증을 나눔 |

각 장은 다음 카테고리 중 하나를 기본값으로 갖습니다. 카테고리는 장의 깊이를 제한하고, 원고 작성 시 어떤 종류의 근거와 예시가 필요한지 판단하는 기준입니다.

| 카테고리 | 의미 | 기본 산출물 |
| --- | --- | --- |
| 개념 정의 | 용어와 범위를 정리하는 장 | 정의, 혼동되는 표현, 짧은 예시 |
| 역사와 패러다임 | 기술 흐름과 관점의 변화를 설명하는 장 | 연표, 주요 전환점, 근거 자료 |
| 모델링 관점 | 현실 문제를 데이터, 특징, 모델, 출력으로 바꾸는 장 | 문제 구조도, 입력/출력 예시 |
| 수학 기초 | AI 계산을 읽기 위한 수학 문법을 복구하는 장 | 최소 수식, 직관, 작은 계산 |
| SW 도구 | Python과 데이터 도구를 재사용하는 장 | 설치/실행 기준, 짧은 코드 |
| 학습 원리 | 학습, 평가, 일반화, 최적화를 설명하는 장 | 핵심 개념, 실패 사례, 체크리스트 |
| 알고리즘 | 특정 모델이나 알고리즘을 다루는 장 | 작동 원리, 장단점, 최소 실습 |
| 실무 휴리스틱 | 경험 규칙을 검증 가능한 출발점으로 다루는 장 | 판단 기준, 위험, 검증 방법 |
| 딥러닝 구조 | 신경망 구성요소와 학습 구조를 다루는 장 | 계산 흐름, 다이어그램, 용어 구분 |
| LLM 핵심 | LLM의 입력, 학습, 생성, 평가를 다루는 장 | 용어 체계, 한계, 근거 자료 |
| 서비스 구조 | AI를 실제 서비스로 연결하는 장 | 아키텍처, 운영 고려사항, 보안 |
| 프로젝트 실습 | 배운 내용을 작동하는 산출물로 검증하는 장 | 코드, 결과, 한계, 회고 |
| 법과 정책 | 윤리, 저작권, 보안처럼 외부 규범을 다루는 장 | 법령/정책 출처, 검토 범위 |
| 전망 | 미래 예측과 논쟁을 다루는 장 | 날짜가 있는 외부 근거, 근거별 구분 |

## 로드맵

### 1장. AI란 무엇인가

역할: AI라는 말의 범위와 혼동을 정리하고, 이후 장에서 다룰 규칙 기반 AI, 머신러닝, 딥러닝, 생성형 AI, LLM의 위치를 잡는 장입니다.

현재 작성 파일:

- `docs/chapters/chapter-01/section-01.md`

상태: 초안

다룰 질문:

- AI라는 말은 왜 문맥에 따라 다르게 쓰이는가
- AI, 머신러닝, 딥러닝, 생성형 AI는 어떻게 연결되는가
- AI를 특정 제품이 아니라 문제를 다루는 분야로 볼 수 있는가
- 표준적인 설명과 개인적인 작업 가설을 어떻게 구분할 것인가

## Part 1. AI 개론과 지형도

역할: 세부 기술을 깊게 들어가기 전에 전체 지형을 다시 잡습니다. "AI란 무엇인가"에서 시작하되, 단순 입문서식 나열이 아니라 이후 파트로 이어지는 연결 지도를 만듭니다.

Part 1은 공개 교재와 대학 강의에서 반복되는 `AI 개론`, `탐색`, `문제 해결`, `불확실성`, `학습` 축을 먼저 반영합니다. 그 위에서 사용자의 직관을 일반화합니다. 특히 "신경망학습, 모델링, 추론의 흐름이 어렵다"는 경험을 다음 질문으로 바꿉니다.

- 사람이 규칙을 쓰는 방식과 데이터에서 규칙을 배우는 방식은 무엇이 다른가
- 문제를 입력과 출력의 관계로 모델링한다는 것은 무엇인가
- 학습은 무엇을 바꾸고, inference는 무엇을 계산하는가
- 가중치는 지식, 패턴, 표현 중 무엇에 가까운가
- 이미지 인식, 객체 검출, 음성 합성, 번역, 텍스트 생성은 같은 딥러닝 흐름의 어떤 변형인가
- LLM은 이 흐름 위에서 무엇을 이어받고, 무엇을 새로 확장했는가

첫 작성 대상:

- LLM 발전사
- 토큰화
- 임베딩
- 다음 토큰 예측과 생성
- RAG

완료 기준:

- LLM이 다음 토큰 예측 모델이라는 설명의 의미와 한계를 말할 수 있다.
- 프롬프트, 임베딩, 벡터 검색, RAG의 관계를 설명할 수 있다.
- Agent가 단순 챗봇과 어떻게 다른지 설명할 수 있다.

## Part 6. 프로젝트

역할: 배운 내용을 작은 산출물로 검증합니다. 프로젝트는 학습 내용을 실제 작동하는 형태로 연결하고, 한계와 개선점을 기록하는 방식으로 작성합니다.

확정 프로젝트:

| 프로젝트 | 카테고리 | 상태 | 검증할 개념 |

| 데이터 분석 미니 프로젝트 | 프로젝트 실습 | 확정 | Pandas, 시각화, 데이터 탐색 |
| 전통 머신러닝 예측 모델 | 프로젝트 실습 | 확정 | 회귀/분류, 평가 지표 |
| 이미지 분류 모델 | 프로젝트 실습 | 확정 | CNN, 학습 루프 |
| 텍스트 분류 모델 | 프로젝트 실습 | 확정 | 토큰화, 임베딩 |

- 데이터 분석 미니 프로젝트
- 문서 기반 RAG 챗봇

완료 기준:

- 문제 정의, 데이터, 접근 방법, 구현, 결과, 한계가 문서에 남아 있다.
- 프로젝트가 앞선 파트의 어떤 개념을 검증하는지 연결되어 있다.
- 실패하거나 부족한 부분도 기록되어 있다.

## 확장 규칙

새 장을 추가할 때는 다음을 확인합니다.

- 어느 Part에 속하는가
- 앞 장의 어떤 개념을 이어받는가
- 뒤 장의 어떤 개념을 준비하는가
- 자체 초안인지, 외부 자료 기반인지
- `mkdocs.yml`의 `nav`에 반영해야 하는가
- 차트나 다이어그램이 필요한가
- 검증 필요 목록에 추가할 질문이 있는가
- 예측성 내용이라면 명확한 외부 근거가 확보되었는가
- 사실 주장마다 근거가 있거나, 근거가 없는 경우 상태가 명확히 표시되었는가
