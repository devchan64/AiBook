# P2-14.1 근거 검토: Git은 변경 이력을 관리하는 도구

## 사용한 주요 자료

| 자료 | 확인한 핵심 | 본문 반영 지점 |
| --- | --- | --- |
| Scott Chacon and Ben Straub, `Pro Git 2nd Edition: About Version Control` | 버전 관리는 시간이 지남에 따라 파일 변화 기록을 남기고 특정 버전을 다시 불러올 수 있게 하는 시스템으로 설명된다. 분산 버전 관리에서는 각 클라이언트가 저장소 이력을 함께 가진다. | Git을 파일 저장 도구가 아니라 변경 이력 관리 도구로 설명 |
| Git project, `git-status Documentation` | `git status`는 작업 트리와 인덱스의 상태를 보여 주며, 커밋할 변경과 추적되지 않은 파일 등을 확인하는 명령이다. | 현재 작업 상태를 묻는 명령으로 설명 |
| Git project, `git-commit Documentation` | `git commit`은 현재 인덱스 내용을 새 커밋으로 기록한다. | 스테이징한 변경을 의미 있는 변경 묶음으로 이력에 남기는 설명 |

## 범위 판단

- P2-14.1은 Git의 기본 목적과 변경 이력 관리 관점에 집중했다.
- 브랜치, 원격 저장소, dev/main 운영, GitHub Pages 배포는 P2-14.2로 넘긴다.
- 충돌 해결, 리베이스, reset, stash 같은 고급 명령은 이 절에서 다루지 않는다.
- Git 내부 객체 모델은 초심자 흐름을 벗어나므로 제외했다.

## 본문 구성 판단

- Part 2의 앞 절들이 코드, 이미지, 원고를 함께 생성했기 때문에 Git을 “학습 산출물의 연결을 추적하는 도구”로 설명했다.
- `working tree -> staging area -> repository` 흐름은 Mermaid로 간단히 표현했다.
- 명령어는 실제 터미널 명령이므로 코드블록으로 유지했다.
- 일반 설명 목록은 코드블록을 쓰지 않고 표와 목록으로 정리했다.

## 조심한 지점

- Git을 GitHub와 같은 서비스로 설명하지 않았다.
- 커밋을 단순 저장으로 설명하지 않고, 의미 있는 변경 묶음으로 설명했다.
- 되돌리기 기능만 강조하지 않고, 왜 바뀌었는지 설명하는 기록 관점으로 정리했다.
- 브랜치와 배포 흐름을 미리 깊게 다루지 않아 P2-14.2의 영역을 남겼다.

## 이 절의 핵심 정리

- P2-14.1의 중심 질문은 “Git은 왜 필요한가?”다.
- 핵심 답은 Git이 파일의 현재 상태만 저장하는 것이 아니라, 의미 있는 변경 묶음을 시간 순서대로 기록해 나중에 다시 읽게 해 준다는 것이다.
- 이 책에서는 Git을 원고, 코드, 이미지, 근거 메모의 관계를 추적하는 학습 기록 도구로 설명한다.
