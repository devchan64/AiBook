# 차트와 다이어그램 작성 기준

이 문서는 책 본문에 들어가는 차트와 다이어그램의 작성 기준을 정리합니다. 특히 수학 개념, 모델 계산, 벡터·행렬, 변화율처럼 시각적 설명이 필요한 도식을 만들 때 참고합니다.

## 기본 원칙

- 차트와 다이어그램은 장식보다 설명 목적을 우선합니다.
- 차트는 본문에서 설명하는 하나의 중심 개념을 시각적으로 보강해야 합니다.
- 한 차트에 너무 많은 개념을 넣지 않습니다. 두 개 이상의 중심 개념이 섞이면 차트를 나눕니다.
- 외부 차트를 그대로 복사하지 않습니다. 필요한 경우 공개 라이선스와 인용 가능성을 확인하고, 가능하면 자체 도식으로 다시 작성합니다.
- 외부 자료의 도식이나 차트를 참고해 재작성한 경우에도 근거 메모에 출처와 재작성 여부를 남깁니다.

## 차트 분리 기준

- 한 차트 안에 서로 다른 시각 언어가 섞이면 차트를 나누는 것을 먼저 검토합니다.
- 좌표축, 곡선, 점, 벡터처럼 위치 자체가 의미인 부분과 박스-화살표 흐름처럼 관계가 의미인 부분은 같은 차트에 억지로 넣지 않습니다.
- 위치가 중요한 부분은 SVG, Matplotlib, Vega-Lite 같은 정밀 차트 도구로 만들고, 단순 관계나 반복 흐름은 Mermaid, PlantUML, D2, Graphviz DOT 같은 다이어그램 도구로 분리합니다.
- 예를 들어 경사하강법을 설명할 때 손실 곡선 위의 이동은 SVG가 적합하고, `current parameters -> gradient -> learning rate -> update` 같은 반복 루프는 Mermaid가 적합합니다.
- 한 그림 안에서 `수학 그래프`와 `프로세스 흐름도`가 함께 필요하다면, 본문에서는 그래프를 먼저 보여 주고 이어서 흐름도를 따로 보여 주는 방식을 우선합니다.
- 독자가 한 번에 읽어야 하는 핵심 질문이 두 개 이상이면 차트를 분리합니다. 예를 들어 “곡선에서 어디로 움직이는가”와 “학습 루프가 어떤 단계로 반복되는가”는 서로 다른 질문입니다.

## 파일 형식과 위치

도식은 무조건 SVG로 만들지 않습니다. 설명 목적에 맞는 텍스트 기반 도구를 먼저 선택합니다.

| 도구 | 적합한 경우 | 주의할 점 |
| --- | --- | --- |
| Mermaid | 단순 흐름도, 개념 관계, 단계, 파이프라인, 상태 전이 | 세밀한 좌표, 곡선, 수학 그래프 표현에는 부적합할 수 있음 |
| PlantUML | UML, 시퀀스 다이어그램, 컴포넌트 관계, 시스템 구조 | 렌더링 환경과 플러그인 지원 여부를 먼저 확인해야 함 |
| D2 | 시스템 아키텍처, 서비스 관계, 네트워크형 구조, 복잡한 박스-화살표 도식 | MkDocs 빌드·배포 파이프라인에서 렌더링 방식이 준비되어야 함 |
| Graphviz DOT | 그래프(graph), 트리(tree), 의존성 관계, 노드-엣지 구조 | 레이아웃은 자동 배치에 맡기는 편이 좋고, 세밀한 위치 조정에는 부적합할 수 있음 |
| Vega-Lite | 통계 차트, 분포, 산점도, 막대그래프, 상호작용 가능한 데이터 시각화 원본 | 정적 배포에서는 SVG 또는 PNG로 내보내는 절차가 필요할 수 있음 |
| Observable Plot | 데이터 기반 차트, 분포, 시계열, 비교 시각화 | JavaScript 실행 환경과 정적 산출물 생성 방식을 확인해야 함 |
| Matplotlib | 수식 그래프, 함수 곡선, 학습 곡선, 실습 코드와 함께 만드는 차트 | 차트 내부 문구와 스타일을 책 기준에 맞춰 후처리해야 할 수 있음 |
| seaborn | 통계적 분포, 회귀선, 범주형 비교처럼 통계 차트가 필요한 경우 | 기본 스타일을 그대로 쓰기보다 책의 색상·라벨 기준에 맞춥니다 |
| Plotly | 상호작용이 필요한 탐색형 차트, 웹 기반 데이터 시각화 | 책 본문에서는 정적 이미지로 대체할지, HTML 상호작용을 허용할지 먼저 결정해야 함 |
| Excalidraw | 개념 스케치, 강의판 느낌의 설명, 초기 설계 초안 | 최종 책 본문에서는 손그림 스타일이 개념 정확성을 흐리지 않는지 확인해야 함 |
| diagrams.net(draw.io) | 복잡한 시스템 구성도, 네트워크, 업무 흐름도 | 원본 `.drawio` 파일을 함께 보관해야 유지보수가 가능함 |
| TikZ/PGF | 논문형 수학 도식, 정밀한 기하·수학 그림 | LaTeX 렌더링 환경이 필요하고 제작 비용이 높음 |
| SVG | 좌표축, 곡선, 벡터, 행렬 변환, 수학적 위치 관계처럼 정밀 배치가 필요한 차트 | 수동 좌표 관리가 필요하므로 겹침과 모바일 표시를 반드시 확인해야 함 |
| PNG | 외부 도구로 만든 최종 이미지, 복잡한 시각 자료 | 원본 수정이 어려우므로 가능하면 생성 원본을 함께 보관해야 함 |

- 단순한 개념 흐름은 Mermaid를 우선 검토합니다.
- UML, 시퀀스, 컴포넌트 관계가 중심이면 PlantUML을 검토합니다.
- 서비스 구조나 복잡한 시스템 관계는 D2를 검토합니다.
- 노드와 엣지 관계가 중심이면 Graphviz DOT를 검토합니다.
- 데이터 분포, 산점도, 막대그래프, 학습 곡선처럼 데이터 기반 차트는 Vega-Lite, Matplotlib, seaborn, Plotly를 검토합니다.
- 손으로 설명하는 느낌이 학습에 도움이 되는 초안은 Excalidraw를 사용할 수 있으나, 최종 본문에서는 가독성과 정확성을 다시 검토합니다.
- 복잡한 아키텍처나 업무 흐름은 diagrams.net을 사용할 수 있으나, 원본 파일을 함께 보관합니다.
- 논문형 수학 도식이나 정밀 기하 도식은 TikZ/PGF를 검토하되, 렌더링 환경과 유지보수 비용을 먼저 따집니다.
- 수학 개념, 모델 계산, 벡터·행렬, 변화율처럼 좌표와 위치가 중요한 설명 차트는 SVG를 사용합니다.
- Markdown 표로 충분한 비교는 차트로 만들지 않습니다.
- 복잡한 그림은 `docs/assets/` 아래에 SVG 또는 PNG로 저장합니다.
- SVG, D2, PlantUML, DOT, Vega-Lite, diagrams.net처럼 생성 원본이 있는 도식은 가능하면 원본을 함께 보관합니다. 수동 SVG라면 SVG 자체가 원본 역할을 합니다.
- 이미지 파일은 해당 본문 위치에 맞춰 `docs/assets/part-XX/chapter-YY/` 아래에 둡니다.

## SVG 기본 구조

- SVG에는 반드시 `<title>`과 `<desc>`를 넣어 접근성과 검색 가능성을 확보합니다.
- 단일 개념 차트는 기본 `viewBox="0 0 760 430"`을 우선 사용합니다.
- 좌우 비교, 전후 비교, 변환 전후처럼 두 패널이 필요한 차트는 `viewBox="0 0 920 430"`을 사용할 수 있습니다.
- 2x2 패널, 여러 카드, 하단 요약 박스를 함께 사용하는 비교 도식은 기본 높이에 맞추려고 억지로 압축하지 않습니다. 패널 내부 도식이 아래로 밀리면 `viewBox` 높이와 패널 높이를 먼저 늘립니다.
- 흐름 화살표, 반복 루프, 하단 요약 박스가 함께 들어가면 `viewBox` 높이를 늘려 충분한 여백을 둡니다.
- 배경은 단순하게 유지합니다.
- 의미 없는 그라데이션, 불필요한 그림자 효과, 장식용 도형을 넣지 않습니다.

## 수학 차트의 구성 요소

수학 차트에는 가능하면 다음 요소를 일관되게 둡니다.

- 제목 또는 핵심 메시지
- 축(axis) 또는 기준선
- 격자(grid) 또는 눈금(tick)
- 주요 점(point), 선(line), 벡터(vector), 구간(interval)
- 변화량 표시: `Δx`, `Δy`, 방향 화살표 등
- 차트 아래 한 문장 요약

모든 요소가 항상 필요하지는 않습니다. 다만 독자가 차트만 보고도 “무엇을 보라는 것인가”를 파악할 수 있어야 합니다.

## 색상 기준

차트의 색상 의미는 일관되게 사용합니다.

| 역할 | 색상 방향 |
| --- | --- |
| 기본 함수, 주 곡선, 주요 벡터 | 파란색 계열 |
| 변화량, 증가, 비교 강조 | 빨간색 계열 |
| 입력 변화, 기준 이동, 보조 방향 | 초록색 계열 |
| 축, 보조선, 흐름 화살표 | 회색 계열 |
| 반복, 되돌아가기, 업데이트 루프 | 파란색 계열 점선 |

색상만으로 의미를 전달하지 않습니다. 라벨, 선 종류, 위치 설명을 함께 둡니다.

- 그래프의 곡선과 동작 화살표가 한 차트에 함께 있으면 같은 색상 계열을 쓰지 않습니다.
- 그래프 곡선은 값이나 함수의 형태를 보여 주고, 화살표는 이동, 반복, 흐름을 보여 주므로 시각 역할을 분리합니다.
- 이 경우 곡선은 청록색, 짙은 중립색, 보라색 등으로 분리하고, 업데이트나 반복 화살표는 파란색 또는 회색 계열로 유지할 수 있습니다.
- 단일 차트 안에서 같은 색은 같은 의미를 가져야 합니다. 같은 파란색이 곡선과 이동 화살표를 동시에 뜻하게 만들지 않습니다.

## 화살표와 흐름 표시

- 차트의 핵심 흐름은 화살표만 보고도 방향을 알 수 있어야 합니다.
- 주요 흐름 화살표는 배경, 카드, 요약 박스와 겹치지 않게 배치합니다.
- 반복 루프나 되돌아가는 화살표는 일반 흐름 화살표보다 굵게 표시하고, 점선이나 색상으로 구분합니다.
- 반복 루프는 하단 요약 문구와 겹치기 쉽기 때문에 요약 박스를 아래로 분리하거나 루프를 별도 여백에 둡니다.
- 화살표 머리는 모바일에서도 보이도록 충분히 크게 둡니다.
- 화살표 머리는 선 두께와 균형을 맞춥니다. 머리만 과하게 크면 흐름보다 장식처럼 보이므로 마커 크기를 줄이거나 선 두께를 함께 조정합니다.
- 화살표가 선, 카드 테두리, 텍스트와 가까우면 의미가 약해지므로 주변에 여백을 둡니다.
- 화살표 머리는 내용 박스 안쪽으로 파고들지 않게 합니다. 박스를 가리키는 화살표는 끝점을 박스 테두리 바깥이나 테두리 직전으로 두고, 머리가 텍스트 영역을 덮지 않는지 확인합니다.
- 화살표가 카드 사이의 관계를 보여 줄 때는 카드 내부 텍스트보다 카드의 바깥 경계나 빈 여백을 향하게 합니다.
- 흐름을 설명하는 라벨은 화살표 바로 위나 아래에 둘 수 있지만, 선과 겹치면 안 됩니다.
- 화살표 라벨이 선과 가까우면 흰색 또는 배경색 캡슐 박스 안에 넣어 독립된 읽기 영역으로 만듭니다.
- 화살표가 굵어질수록 라벨은 더 멀리 두거나 별도 박스로 분리합니다.
- 되돌아가는 루프는 본문 요약 박스 위를 통과하지 않게 하고, 요약 박스와 최소한 한 줄 높이 이상의 간격을 둡니다.
- 좌우 패널을 연결하는 전환 화살표는 라벨과 따로 떨어뜨려 두지 않습니다. `same data`, `convert`, `then` 같은 전환 라벨은 화살표 위의 캡슐 박스 또는 화살표 시작점 근처의 한 묶음으로 배치해 독자가 라벨을 어느 흐름에 붙여 읽어야 하는지 알 수 있게 합니다.
- 좌우 패널 사이의 중앙 전환 영역은 너무 넓게 비워 두지 않습니다. 패널 간격이 넓으면 전환 라벨, 짧은 화살표, 방향 표시를 중앙 정렬해 두 패널의 관계가 끊겨 보이지 않게 합니다.

## 겹침 방지 기준

- 텍스트는 선, 화살표, 카드 테두리, 아이콘, 데이터 점과 겹치면 안 됩니다.
- 카드 내부 텍스트는 카드 테두리와 충분한 안쪽 여백을 둡니다.
- 패널형 차트에서는 모든 내부 도형, 선, 노드, 라벨이 패널 경계 안에 들어와야 합니다. 내부 요소가 패널 밖으로 삐져나오면 패널 높이를 늘리거나 내부 도식을 위로 재배치합니다.
- 패널형 차트에서는 제목과 설명문이 있는 상단 헤더 영역을 별도로 보호합니다. 노드, 선, 카드, 표 같은 도식 요소는 패널 제목과 설명문 아래에 충분한 간격을 두고 시작해야 합니다.
- 행렬, 표, 배열처럼 축(axis) 또는 방향 화살표를 함께 설명하는 도식에서는 축 화살표와 축 라벨까지 패널 내부 요소로 계산합니다. 표 본체만 패널 안에 맞추지 말고, 축 라벨의 실제 하단 위치와 공통 요약 박스 사이에 여백을 확보합니다.
- 축 라벨이 패널 아래쪽에 놓이면 공통 하단 요약 박스와 쉽게 겹칩니다. 이 경우 요약 박스를 아래로 내리거나 `viewBox`와 패널 높이를 늘려 축 라벨을 패널 내부에 완전히 포함시킵니다.
- 축 라벨은 표, 행렬, 강조 셀과 겹치지 않게 배치합니다. 긴 문장형 라벨보다 `axis 0: rows`, `axis 1: columns`처럼 짧은 라벨을 우선하고, 필요하면 배경 캡슐을 둡니다.
- 축이나 전환 화살표의 라벨은 화살표와 시각적으로 붙어 있어야 합니다. 라벨이 화살표에서 떨어져 보이면 캡슐 박스를 화살표 바로 위나 아래에 두고 중앙을 맞춥니다.
- 축 방향 계산 결과의 `result shape` 같은 결과 라벨은 결과 패널 내부에 둡니다. 패널 밖으로 밀려나면 라벨을 억지로 아래에 두지 말고 패널 높이와 `viewBox`를 늘립니다.
- 표, 행렬, 배열의 행·열 라벨과 마지막 행은 하단 요약 박스에 가려지면 안 됩니다. 데이터 본체가 패널 하단에 가까우면 패널과 `viewBox`를 먼저 늘립니다.
- 코드 표현과 설명 문장을 같은 캡슐 안에 넣을 때는 충분한 가로 폭이 확인된 경우에만 같은 줄을 사용합니다. 코드가 길면 코드와 설명을 두 줄로 쌓아 겹침을 방지합니다.
- 슬라이스, 범위, 경계값처럼 아래쪽 주석이 많은 도식은 주석까지 패널 내부 요소로 봅니다. `kept`, `stop before`, `step by` 같은 라벨과 하단 요약 박스 사이에 충분한 간격을 둡니다.
- 패널 안에 `Question: ...` 같은 설명문이 있으면, 원형 노드나 연결선이 해당 문장 위로 올라오지 않게 합니다. 패널 안쪽에서 헤더 영역과 도식 영역을 시각적으로 분리합니다.
- 트리(tree)나 그래프(graph)처럼 노드-엣지 구조가 들어가는 패널은 원형 노드 반지름과 하단 라벨 높이를 포함한 실제 bounding box를 계산해 패널 안쪽 여백을 확보합니다.
- 좌우 비교 차트에서는 양쪽 패널의 내부 도식 중심이 비슷한 높이와 밀도로 보이게 맞춥니다. 한쪽 도식이 지나치게 오른쪽이나 아래쪽으로 치우치면 비교보다 배치 불균형이 먼저 보입니다.
- 2x2 비교 도식에서는 상단 패널과 하단 패널의 시각 밀도를 맞추되, 하단 패널의 도식이 요약 박스와 가까워지면 요약 박스를 아래로 내리거나 `viewBox` 높이를 늘립니다.
- 차트 하단 요약 문구는 도식의 일부가 아니라 해석 문장입니다. 흐름선, 루프, 화살표 머리와 분리합니다.
- 하단 요약 박스는 마지막 패널 또는 마지막 도식 요소와 최소 24px 이상의 수직 간격을 둡니다.
- 전체 차트를 요약하는 문구는 특정 패널 내부에 넣지 말고, 가능하면 패널 밖의 공통 하단 요약 박스로 분리합니다.
- 패널 내부에 보조 설명 박스를 둘 때는 같은 패널의 코드 박스, 표, 노드-엣지 도식과 최소 24px 이상의 간격을 둡니다. 간격을 확보하기 어렵다면 보조 설명을 본문이나 전체 하단 요약으로 옮깁니다.
- 패널 내부 보조 설명 박스의 문구는 짧게 유지합니다. 긴 해석 문장은 패널 밖 공통 요약, 이미지 앞뒤 본문, 또는 체크리스트로 이동합니다.
- 노드-엣지 도식에서 엣지 위 숫자 라벨은 선 위에 바로 얹지 말고 배경 박스와 함께 선에서 약간 떨어뜨립니다. 숫자가 노드 테두리나 다른 엣지와 가까우면 라벨 위치를 조정해 숫자와 연결선의 역할을 분리합니다.
- 코드처럼 줄 단위 텍스트가 들어가는 박스는 마지막 줄이 테두리에 닿지 않도록 아래쪽 내부 여백을 둡니다. 닫는 중괄호, 괄호, 마지막 코드 줄이 박스 경계선과 겹쳐 보이면 박스 높이를 늘리거나 텍스트 y 좌표를 조정합니다.
- 강조 박스, 라벨 박스, 요약 박스가 서로 붙어 있으면 독자는 하나의 요소로 오해할 수 있으므로 시각적 간격을 둡니다.
- 텍스트가 선 위에 올라가면 선을 옮기는 것을 우선하고, 어려우면 텍스트에 배경 박스를 둡니다.
- 한 줄 라벨이 길어져 화살표나 도형과 충돌하면 문구를 줄이거나 두 줄로 나눕니다.
- 차트 안에서 가장 중요한 흐름은 가장 먼저 보이도록 하고, 보조 설명은 별도 박스나 본문 설명으로 넘깁니다.

## 글자와 라벨

- SVG 안의 텍스트는 모바일에서도 읽히도록 너무 작게 만들지 않습니다.
- 일반 라벨은 14px 이상, 제목은 18px 이상을 기준으로 합니다.
- 화살표 라벨, 요약 문구, 카드 내부 설명이 서로 겹치지 않도록 세로 여백을 먼저 확보합니다.
- 한 도표 안에서는 텍스트 정렬 규칙을 먼저 정하고 일관되게 적용합니다.
- 패널 제목, 설명 문장, 긴 문구는 좌측 정렬을 기본으로 합니다.
- 작은 카드, 원, 버튼형 박스처럼 짧은 상태값이나 후보명을 담는 요소는 가운데 정렬을 사용할 수 있습니다.
- 가운데 정렬을 사용할 때는 눈대중으로 `x` 좌표를 보정하지 말고, 요소의 중앙 좌표와 `text-anchor="middle"`을 사용합니다.
- 같은 성격의 요소는 같은 정렬 방식을 사용합니다. 예를 들어 후보 카드가 가운데 정렬이면 선택 결과 카드와 거절 카드도 가운데 정렬합니다.
- 좌측 정렬 요소와 가운데 정렬 요소가 한 차트에 함께 있을 수는 있지만, 역할이 달라야 합니다. 설명문은 좌측 정렬, 짧은 라벨은 가운데 정렬처럼 구분합니다.
- 차트 내부의 보이는 문구는 영어를 기본으로 작성합니다.
- 한국어 설명은 본문에서 차트 앞뒤 문장, 이미지 대체 텍스트, `<desc>`, 근거 메모로 보조합니다.
- 수학 기호, 함수식, 표준 용어는 영어 또는 기호를 함께 사용합니다.
- 본문에서 이미 설명한 용어라도 차트 안에서는 짧은 라벨을 붙입니다.

## 본문 삽입 기준

- 본문에서 차트를 삽입할 때는 이미지 바로 앞뒤에 차트가 무엇을 보여 주는지 짧게 설명합니다.
- 차트는 본문을 대체하지 않습니다. 본문 설명을 보강하는 역할로 사용합니다.
- 차트가 본문 주장에 근거로 쓰이면 근거 메모에 차트의 목적, 자체 제작 여부, 외부 자료 참조 여부를 남깁니다.

## 검증

- 새 차트를 추가한 뒤에는 `mkdocs build`로 링크와 렌더링 오류가 없는지 확인합니다.
- 모바일 화면에서 라벨이 너무 작거나 잘리지 않는지 확인합니다.
- 수식, 축, 화살표가 본문 설명과 모순되지 않는지 확인합니다.
- SVG를 직접 브라우저에서 열어 화살표, 라벨, 요약 박스가 겹치지 않는지 확인합니다.
- MkDocs 페이지에 삽입된 상태에서도 확인합니다. 단독 SVG와 본문 삽입 상태의 표시 크기가 다를 수 있습니다.
- 차트를 수정한 뒤에는 다음 항목을 눈으로 확인합니다.
  - 화살표 방향이 명확한가
  - 화살표 머리가 보이는가
  - 라벨이 선이나 화살표와 겹치지 않는가
  - 하단 요약 문구가 도식 요소와 분리되어 있는가
  - 모바일 폭에서도 텍스트가 잘리거나 겹치지 않는가
