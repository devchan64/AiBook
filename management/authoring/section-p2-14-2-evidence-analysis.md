# P2-14.2 근거 검토: 브랜치, 커밋, 문서 재현성

## 사용한 주요 자료

| 자료 | 확인한 핵심 | 본문 반영 지점 |
| --- | --- | --- |
| Pro Git, `Branches in a Nutshell` | Git 브랜치는 커밋을 가리키는 가벼운 포인터로 설명되며, Git의 브랜칭은 가볍고 자주 분기·병합하는 흐름을 장려한다. | 브랜치를 작업 흐름을 분리하는 이름 붙은 이력으로 일반화 |
| Git project, `git-branch Documentation` | 브랜치 목록, 생성, 삭제, 이름 변경 등 브랜치 조작 명령을 제공한다. | 브랜치를 별도 작업 흐름으로 다루는 설명의 근거 |
| GitHub Docs, `What is GitHub Pages?` | GitHub Pages는 GitHub 저장소에서 정적 사이트를 호스팅하는 서비스로 설명된다. | `main` 반영을 배포 흐름과 연결하는 설명 |

## 범위 판단

- P2-14.1은 Git의 기본 목적과 커밋 개념을 설명했다.
- P2-14.2는 이 책의 `dev`/`main` 운영과 문서 재현성 관점에 집중했다.
- 충돌 해결, rebase, pull request 리뷰, GitHub Actions 세부 설정은 제외했다.
- `main` 반영은 저장이 아니라 배포 실행으로 볼 수 있음을 프로젝트 운영 기준으로 명시했다.

## 조심한 지점

- 브랜치를 Git 내부 포인터로만 설명하지 않고, 초심자가 이해할 수 있는 작업 흐름 분리 관점으로 일반화했다.
- GitHub Pages 설명은 공식 문서의 정적 사이트 호스팅 개념까지만 사용했다.
- 이 저장소의 `dev`/`main` 운영 기준은 외부 일반 규칙이 아니라 프로젝트 내부 규칙으로 구분했다.

## 이 절의 핵심 정리

- 중심 질문은 “작성 중 문서와 배포 문서를 어떻게 구분하고 재현 가능하게 관리할 것인가?”다.
- 핵심 답은 `dev`에서 작성하고 `main`에서 배포하며, 커밋은 문서·코드·이미지·근거의 의미 있는 변경 묶음이어야 한다는 것이다.
